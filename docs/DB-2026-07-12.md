# Trabalho de banco de dados (Supabase) — 2026-07-12

Com acesso à Management API (token no `.env`), auditei o banco real de produção
e apliquei as migrações que faltavam. O schema agora está versionado no repo em
`supabase/`.

## O que foi auditado (antes só existia no dashboard)

- **22 tabelas**, todas com RLS **ligado**.
- **~73 policies** RLS — snapshot em `supabase/schema/policies.sql`.
- **19 funções/RPCs** (SECURITY DEFINER) — `supabase/schema/functions.sql`.
- **13 triggers**, **8 enums** — `supabase/schema/{triggers,types}.sql`.
- FKs, constraints, colunas — `supabase/schema/tables.sql`.

## Bugs de banco corrigidos

### Migration 0001 — `delete_own_account`
Não existia. A exclusão de conta (obrigatória na Play Store / LGPD) só tinha o
fallback por email. Criei o RPC tratando todos os FKs `RESTRICT` (comments,
clubs.criador_id, book_suggestions, voting_rounds, member_removals) e o
invariante de super_admin. Roda como transação atômica: qualquer falha reverte
tudo. Verificado: função criada, guard de autenticação funcionando.

### Migration 0002 — aviso de "livro finalizado"
O app tentava avisar os membros quando um livro era concluído, mas usava
`tipo="book_finished"` que **não existia no enum** e a tabela `notifications`
**não tem INSERT policy** pro cliente — falhava duas vezes, ninguém era avisado.
Corrigi movendo pra um **trigger server-side** (`club_books_notify_finished`)
que cria os avisos quando o livro vira `finished`. À prova de exceção (nunca
bloqueia a conclusão). Removi o insert quebrado do app e adicionei o render.

## O que a auditoria CONFIRMOU estar OK (não eram bugs)

- **Forja de notificação**: a tabela `notifications` **não tem INSERT policy**,
  então o cliente não consegue criar/forjar aviso nenhum. Os avisos legítimos
  vêm de RPCs SECURITY DEFINER (`close_voting_round`, `promote_member`,
  `remove_member`, `transfer_super_admin`) e do novo trigger. A preocupação da
  auditoria já estava mitigada pelo schema.
- **PK de `votes`**: no servidor é `(voting_round_id, user_id)` — correta (1
  voto por rodada). O bug de PK é só no Room **local** (migração de cliente, à
  parte).
- **Editar próprio comentário**: a policy de UPDATE de `comments` já permite o
  autor editar — é só questão de UI no app (não precisa mexer no banco).
- **Enumeração por código / clubes arquivados**: join é via RPC; RLS de `clubs`
  é "membros ou público". Sem vazamento de clube privado.
- **Segurança das funções**: todas as 19 SECURITY DEFINER têm `search_path`
  fixo (sem risco de escalonamento).

## Deferido (decisão de produto, não bug)

- **T2 — recorrência de encontro não gera encontros automaticamente.** O padrão
  ("toda segunda 19h") é salvo mas nenhum encontro é criado a partir dele.
  Implementar auto-geração exige lógica de data pros 6 tipos de recorrência +
  decisões (quantos à frente, fuso, evitar duplicado). Um trigger de data-math
  errado criaria encontros errados em produção — pior que o estado atual (admin
  cria manualmente). Fica pra implementar com o app rodando pra verificar cada
  tipo de recorrência.
- Tipos de aviso `comment_on_chapter` / `voting_open` / `meeting_reminder` são
  valores do enum **sem produtor** — features nunca implementadas. Precisam de
  decisão de produto (quem recebe, com que frequência, evitar spam) antes de
  criar os triggers.

## Como aplicar/reproduzir

As migrações em `supabase/migrations/` foram aplicadas via Management API
(project ref no `SUPABASE_URL`). Para reaplicar num ambiente novo, rodar os
arquivos em ordem. O snapshot em `supabase/schema/` reflete o estado atual.
